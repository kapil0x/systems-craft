Phase 1: File-Based Partitioned Queue

Goal: Understand message queue fundamentals through a simple file-based implementation.

What You'll Build:

  • 4-partition queue with hash-based routing (client_id ‚Üí partition)
  • Sequential offset management per partition
  • Producer: enqueue(client_id, data) writes to queue/partition-N/OFFSET.msg
  • Consumer: QueueConsumer reads from partitions with offset tracking

Performance Results:

  • Throughput: ~800 RPS (fsync bottleneck)
  • Latency: 0.70ms avg (file I/O bound)
  • Success Rate: 97.4% @ 50 clients
  • Limitation: Single machine only, disk I/O bound

Key Learning: File-based queues are simple and durable, but limited by disk I/O. Production systems need batching and async I/O to scale.

‚ö†Ô∏è 4 Critical Threading Bugs Fixed

Bug #1: Race condition - 16 HTTP workers accessing 1 shared KafkaProducer without mutex

Bug #2: Use-after-free in destructor when messages still in-flight during shutdown

Bug #3: No retry logic for ERR__QUEUE_FULL errors (drops messages)

Bug #4: Message lifetime issues with async sends (stack corruption)

Key Learning: librdkafka is NOT thread-safe. Requires external mutex synchronization.

Architecture Insight: 16 HTTP worker threads ‚Üí 1 shared KafkaProducer (with mutex) ‚Üí 1 background I/O thread ‚Üí Kafka broker

Key Learnings:

  • Batching: Kafka batches messages for efficiency (vs. file-based per-message fsync)
  • Async I/O: Zero-copy transfers and memory-mapped buffers
  • Thread Safety: Defense in depth: flush(10s) + poll() loop + warnings
  • Scalability: Horizontal scaling vs. single-machine limitations

Documentation: Full analysis in docs/kafka-fix/OVERVIEW.md
