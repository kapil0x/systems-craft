The 5 Components

1. Ingestion API

HTTP POST /metrics endpoint accepting JSON payloads. Single-threaded, blocking request handling.

Tech: Raw sockets, simple HTTP parsing

2. In-Memory Queue

Thread-safe queue buffering metrics between ingestion and storage. Unbounded, no persistence.

Tech: std::queue + std::mutex

3. Storage Consumer

Background thread polling queue and writing metrics to file. Blocking I/O, append-only.

Tech: std::thread, std::ofstream

4. Query API

HTTP GET /query endpoint reading from file and filtering by name and time range. Full file scan.

Tech: Linear scan, in-memory filtering

5. Alerting Engine

Background thread checking alert rules periodically. Calculates averages, compares to thresholds.

Tech: Polling model, console logging

ğŸ“ What You'll Learn

Core Concepts

  • Component Boundaries: How to design clean APIs between system components
  • Producer-Consumer Pattern: Thread-safe queues for decoupling producers from consumers
  • HTTP API Design: Building RESTful endpoints for metrics ingestion and querying
  • Time-Series Storage: JSON Lines format for append-only metric storage
  • Rule-Based Alerting: Evaluating thresholds over time windows

Performance Understanding

  • Bottleneck Identification: See single-threaded ingestion limiting throughput to ~50 RPS
  • Blocking I/O Impact: Understand how file writes create backpressure
  • Linear Scan Costs: Feel the slowdown of O(n) queries without indexes
  • Polling Latency: Experience 10-second alert delays from fixed check intervals

âš ï¸ Intentional Bottlenecks

This system is deliberately slow. The single-threaded ingestion, blocking I/O, and full file scans are educational bottlenecks. You'll feel the pain, measure it with benchmarks, then fix it in Crafts #1-5. This is the learning method.
